هدف این اصلاح: بانک ترخیصان دوباره از پرداخت‌های واقعی شما پر شود، دفتر کل خالی نماند، و مشکل اصلی نمایش/ویرایش پرداختی‌های کارت هم باقی نماند.

### 1) برگرداندن منبع حقیقت بانک ترخیصان به داده‌های واقعی
- موجودی، کل ورودی و دفتر کل فقط وابسته به `ts_treasury_ledger` خالی نباشد.
- اگر دفتر کل خالی یا ناقص بود، API خودش از روی پرداخت‌های تأییدشده‌ی کاربران (`ts_card_payments`) ورودی‌های صندوق را محاسبه کند.
- خروجی صندوق فقط پرداخت‌های کارت‌هایی باشد که واقعاً `from_treasury=1` دارند یا در دفتر کل با منبع `admin_payment` ثبت شده‌اند.
- سود نقدی همان اختلاف دریافتی کاربران و پرداختی کارت‌ها بماند، ولی موجودی بانک با ورودی/خروجی واقعی هم‌خوان شود.

### 2) بازسازی خودکار دفتر کل از پرداخت‌های موجود
- در `treasury-ledger.php` اگر ledger خالی/ناقص بود، ردیف‌های قابل نمایش را از پرداخت‌های واقعی بسازد و برگرداند.
- در `treasury-summary.php` هم اگر ledger خالی بود، آمار بانک را از همین fallback بدهد تا «ورودی» و «دفتر کل صندوق» خالی نشوند.
- یک helper مشترک در `db.php` اضافه می‌شود که ledger را از پرداخت‌های قدیمی backfill/sync کند، بدون اینکه داده‌ی تکراری بسازد.

### 3) اصلاح install.php برای هاست
- بعد از آپلود بیلد، اجرای `api/install.php` باید فقط ساختار را کامل کند و همچنین پرداخت‌های قبلی را به دفتر کل sync کند.
- خطاهای Duplicate column دیگر خروجی را خراب نکند.

### 4) حفظ مشکل اصلی پرداختی‌های کارت
- نمایش تاریخ، فیش و یادداشت در `card-admin-payments-list.php` حفظ و پایدار می‌شود.
- ویرایش وضعیت پرداخت کارت (`confirmed / pending / rejected`) ledger بانک را sync می‌کند:
  - اگر از confirmed خارج شود، اثرش از بانک حذف می‌شود.
  - اگر confirmed شود و منبع پرداخت «بانک ترخیصان» باشد، خروجی بانک ثبت می‌شود.
- منبع پرداخت بعد از ثبت همچنان قفل می‌ماند و فقط همان گزینه‌ی اصلی فعال است.

### 5) تست کم‌ریسک قبل از تحویل
- PHP lint روی فایل‌های API تغییرکرده اجرا می‌شود.
- منطق SQL طوری نوشته می‌شود که اگر ledger موجود باشد خرابش نکند، و اگر خالی باشد از پرداخت‌های واقعی بازسازی کند.

بعد از تأیید شما، همین اصلاحات محدود و نهایی را اعمال می‌کنم.
